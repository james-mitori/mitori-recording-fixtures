Labeling guide

Describe what was visible, not what should have happened. The examples below use the same buttons and pills you'll see in the right pane.

Each step has these required pieces

Start & end Press A at the visible start, S when it ends. Click any timeline block to jog the video back to that step.
Action What the worker visibly did. Click a starter chip to drop a verb in: Clicked Typed Selected
Don't write "tried to" or "wanted to".
Action types Tag the kinds of input — used to cross-check with telemetry. click keystroke scroll
Screen / app The visible app, page, or screen. "Salesforce account record", "Excel: Invoices.xlsx".
Evidence Quote what's on screen — a field value, message, button label, page state. "Toast: 'Saved'". Not a restatement of the action.
Confidence How sure you are based on what's visible.

Per-step exception types

Toggle Mark this step as an exception when the step is messy work, then pick the closest type.

Interruption Unrelated activity pulls the worker away from this workflow. Phone call, chat ping, email check, switching to a different task. The workflow pauses or detours, then comes back.
Rework The worker did something then visibly undid or redid it. Typed a wrong value and deleted, copied the wrong cell, hit undo, retried.
Mistake A visible error that wasn't fixed. Clicked the wrong button, entered the wrong account, sent to the wrong recipient.
Abandoned The worker started something and never completed it. Closed the tab, gave up, switched away permanently.
Other Anything else messy that doesn't fit. Add a short reason in the details box.

Per-step exception vs whole-video Notes drawer

If you can put it on a step, put it on a step. The Notes drawer (right edge, key N) is only for things that don't fit a single step.

  • Observed discrepancies — visible mismatches between sources. Spreadsheet shows £400, CRM shows £450. About the data, not the worker.
  • Whole-video uncertainty — things you couldn't observe. "Couldn't read the toast in the bottom-right; cursor blocked it." About what you, the labeler, couldn't see.

Granularity — when to split a step

Copy and paste Two steps when both are visible. Copy is one observable action (highlight + Cmd+C in source); paste is another (Cmd+V in destination, value appears). Different timestamps, different evidence. If only one is visible — e.g. you see the paste happen but never see the copy — label only that one.
Multiple field updates One step per field. Updating Account Owner, Renewal Date, and Billing Email is three steps, not one "updated record" step. The benchmark grades the AI on each field separately.
Searching for a record One step. Typing into the search box and clicking the result are usually one logical action — searching for "Acme North." Split only if there's a meaningful pause or a different visible state in between.
Click then wait One step. A click that triggers loading is a single action. The wait is part of it. Use action types click + wait to capture both.

Common mistakes — read these once

  • Writing intent ("tried to save", "wanted to update") — describe what was visible instead.
  • Saying the action and evidence as the same thing — evidence should describe what's on screen, not what the worker did.
  • Hiding interruptions or mistakes to keep the label "clean" — these are exactly what the benchmark needs.
  • Using a confirmation message as proof of back-end state — only describe what's visible.
  • Leaving steps with "—" as start or end — every step needs both timestamps.
  • Using vague evidence like "yes" or "done" — quote the field value, page text, or button label.
